# avito_ds_bootcamp
## Результат

Лучший результат на публичном leaderboard:

```text
MAP@10 = 0.554196
```

Он получен моделью LambdaMART


## Основные выводы EDA

### Запросы

Пользовательские запросы короткие, разговорные и часто содержат:

- опечатки;
- разные словоформы;
- неформальные формулировки;
- вежливые вводные слова;
- сокращённое описание проблемы без точного названия раздела справки.

Поэтому в решении используются одновременно:

- word-level признаки;
- char-level признаки;
- лемматизированные тексты;
- поиск по чанкам статей;
- supervised-сигналы из calibration.

### Статьи

Статьи представлены HTML-документами и сильно различаются по длине. Некоторые документы очень большие, поэтому поиск только по полному тексту размывает локально релевантные фрагменты.

Для этого статьи разбиваются на чанки, а score статьи вычисляется по наиболее релевантным фрагментам.

### Дубли

В данных встречаются статьи с одинаковым или почти одинаковым содержанием.

Найдены:

- 6 групп полностью одинакового HTML;
- 36 групп с одинаковым очищенным текстом;
- отдельные near-duplicate пары и случаи, когда короткая статья почти полностью содержится в длинной.

При этом calibration содержит пример, где два полностью одинаковых документа одновременно входят в ground truth. Поэтому `article_id` нельзя удалять из пространства кандидатов.

Безопасная стратегия:

1. одинаковые тексты можно индексировать совместно;
2. все исходные `article_id` сохраняются;
3. score документа раскрывается обратно на все идентификаторы группы;
4. точные дубли добавляются в candidate set reranker-моделей.

Near-duplicate статьи принудительно объединять не использовалось: прирост на локальной проверке оказался пренебрежимо мал.

## Предобработка

Основная нормализация:

- преобразование HTML в обычный текст;
- lowercase;
- замена `ё` на `е`;
- удаление пунктуации;
- схлопывание пробелов;
- удаление русских стоп-слов и вежливых вводных конструкций.

Дополнительно используется лемматизация через `pymorphy3`.

Пример:

```text
объявления, объявлению, объявлений → объявление
отправил, отправить, отправляю → отправить
```

Лемматизация используется как отдельный retrieval-канал. Исходный текст сохраняется для char TF-IDF и transformer-моделей.

## Ноутбуки

### `eda_avito.ipynb`

Полный разведочный анализ данных.

Содержит:

- проверку структуры файлов;
- статистики длины запросов и статей;
- анализ HTML-разметки;
- частотный анализ слов и биграмм после лемматизации;
- распределение количества релевантных документов;
- частоты `article_id` в calibration;
- анализ дубликатов и near-duplicate статей;
- сравнение calibration и test;
- диагностические retrieval-бейзлайны.

---

### `lexical_baseline.ipynb`

Базовые лексические методы:

- BM25;
- word TF-IDF;
- char TF-IDF;
- Reciprocal Rank Fusion.

Лучшим вариантом внутри этого этапа оказался BM25-only.

Публичный результат:

```text
MAP@10 = 0.314717
```

Ноутбук служит прозрачным baseline и точкой сравнения для более сложных решений.

---

### `chunk_retrieval.ipynb`

Поиск по фрагментам статей.

Используются:

- char TF-IDF по исходному тексту;
- word TF-IDF по лемматизированному тексту;
- агрегация chunk-level score до уровня статьи;
- Reciprocal Rank Fusion.

Публичный результат:

```text
MAP@10 = 0.390781
```

Модель хорошо переносится с calibration на test и практически не показывает переобучения.

---

### `calibration.ipynb`

Поиск похожих размеченных запросов из calibration.

Similarity строится как комбинация:

- raw word TF-IDF;
- char TF-IDF;
- lemmatized word TF-IDF.

Голоса ближайших calibration-запросов переносятся на связанные с ними `article_id`.

Локальный результат:

```text
MAP@10 ≈ 0.6238
```

Публичный результат:

```text
MAP@10 = 0.542795
```

Сильная модель, но локальная leave-one-out оценка заметно переоценивает переносимость на скрытый test.

---

### `hybrid.ipynb`

Объединение:

- chunk retrieval;
- calibration kNN;
- Reciprocal Rank Fusion.

Локальный результат:

```text
MAP@10 ≈ 0.6259
```

Публичный результат:

```text
MAP@10 = 0.533297
```

Hybrid оказался слабее чистого kNN на leaderboard. Дополнительный lexical-канал иногда нарушает хороший порядок документов, найденных через calibration.

---

### `lambdamart.ipynb`

Learning-to-rank решение на LightGBM LambdaMART.

Используемые признаки:

- title word TF-IDF score;
- title char TF-IDF score;
- title lemma TF-IDF score;
- chunk char score;
- chunk lemma score;
- calibration kNN score;
- rank каждого retrieval-канала;
- длина статьи;
- количество чанков;
- размер duplicate-группы;
- частота статьи в calibration;
- число соседей, проголосовавших за статью.

Candidate set строится как union нескольких retrieval-источников. Fold-dependent признаки рассчитываются только на train-части каждого fold, чтобы не допустить прямой утечки разметки.

Локальный 3-fold OOF результат:

```text
MAP@10 ≈ 0.6518
```

Публичный результат:

```text
MAP@10 = 0.554196
```

Это лучший отправленный вариант.

## Главный вывод по валидации

Лексические модели показали почти одинаковое качество локально и на leaderboard:

```text
BM25: 0.316 → 0.315
Chunks: 0.388 → 0.391
```

Supervised-модели, использующие calibration, просели заметнее:

```text
kNN: 0.624 → 0.543
Hybrid: 0.626 → 0.533
LambdaMART: 0.652 → 0.554
```

Это указывает на различие распределений calibration и test и на переобучение к небольшому набору из 500 размеченных запросов.

Практический вывод:

- calibration-сигналы очень сильные, но требуют осторожной валидации;
- lexical retrieval хуже по абсолютному качеству, зато стабильнее переносится;
- dense и cross-encoder модели без доменного дообучения не гарантируют прироста;
- дальнейшее улучшение стоит искать в более устойчивом сочетании supervised и corpus-based retrieval, а не в простом усложнении модели.

## Возможные улучшения

1. Более строгая group-aware или temporal-style валидация вместо случайных fold.
2. Domain adaptation dense-модели на парах из calibration.
3. Hard-negative mining для cross-encoder.
4. Query expansion по заголовкам и структуре справочного центра.
5. Отдельная модель оценки количества релевантных документов.
6. Distillation устойчивого lexical retrieval в learning-to-rank модель.
7. Использование структуры HTML и заголовков разделов как отдельных полей.
8. Калибровка веса calibration-prior для снижения переобучения.
